    38тЖТ## 1. API Philosophy
    39тЖТ
    40тЖТ| Principle | Rule | Why |
    41тЖТ|---|---|---|
    42тЖТ| **BFF owns the contract** | The frontend never calls Medusa directly. All requests go through Next.js BFF routes. | Shield the frontend from Medusa API changes. Add GGH-specific logic (delivery, pricing) without coupling. |
    43тЖТ| **Minimal payloads** | Every response includes only what the UI needs. No `SELECT *` leaking into JSON. | Elder users on slow mobile connections. Smaller payloads = faster loads. |
    44тЖТ| **Arabic-first, bilingual always** | API responses include both `name_en` and `name_ar`. The frontend picks based on locale. No server-side locale switching. | RTL/LTR is a presentation concern. The API serves data; the UI renders it. |
    45тЖТ| **Predictable structure** | Every response follows the same envelope. Every error follows the same shape. | Om Ibrahim should never see a raw stack trace. The frontend should never need `if (error.type === ...)` spaghetti. |
    46тЖТ| **Idempotent by default** | `PUT` and `DELETE` are idempotent. `POST` for creation returns the created resource. `PATCH` for partial updates. | Accidental double-taps (elder users) must not create duplicate orders. |
    47тЖТ| **Money as integers** | All monetary values in API payloads are piastres (integer). The frontend formats to EGP. | Same reason as the database: no floating-point errors. EGP 110.50 тЖТ `11050`. |

   108тЖТ## 3. URL Structure & Versioning
   109тЖТ
   110тЖТ### 3.1 Version Strategy
   111тЖТ
   112тЖТ| Decision | Choice | Rationale |
   113тЖТ|---|---|---|
   114тЖТ| Versioning style | URL path prefix (`/api/v1/`) | Explicit, cacheable, elder-friendly тАФ no hidden headers to explain |
   115тЖТ| Current version | `v1` | First production release |
   116тЖТ| When to bump | Breaking changes only. Additive changes (new fields, new endpoints) do not bump. | Follow SemVer principles. Clients must ignore unknown fields. |
   117тЖТ| Max supported versions | 2 concurrent (`v1`, `v2` during migration) | We deprecate old versions with 6-month sunset notice. |
   118тЖТ| Medusa API versioning | Handled internally by BFF. Frontend never sees Medusa version numbers. | BFF absorbs Medusa version changes. |
   119тЖТ

   146тЖТ### 3.3 URL Naming Rules
   147тЖТ
   148тЖТ| Rule | Convention | Example |
   149тЖТ|---|---|---|
   150тЖТ| Resource names | Plural nouns | `/products`, `/orders`, `/cart/items` |
   151тЖТ| Nested resources | Maximum 2 levels | `/orders/:id/items` тЬУ тАФ `/orders/:id/items/:itemId/variants` тЬЧ |
   152тЖТ| Actions as verbs | Only when not CRUD | `/cart/checkout`, `/auth/login`, `/auth/logout` |
   153тЖТ| No verbs in CRUD | Use HTTP method | `POST /cart/items` (not `POST /cart/add-item`) |
   154тЖТ| Kebab-case for multi-word | `delivery-zones`, `price-tiers` | `/delivery-zones/:id/slots` |
   155тЖТ| IDs in path | Only for single-resource operations | `/products/prod_01ABC` |
   156тЖТ| Query for filtering | Never in path | `/products?category=rice&in_stock=true` |

   195тЖТ### 4.2 Session Strategy
   196тЖТ
   197тЖТ| Property | Value | Why |
   198тЖТ|---|---|---|
   199тЖТ| Token type | HttpOnly cookie (`ggh_session`) | XSS-safe. No token in localStorage. |
   200тЖТ| Cookie attributes | `Secure; SameSite=Strict; Path=/; Max-Age=86400` | 24-hour session. Strict CSRF protection. |
   201тЖТ| Token format | JWT (signed, not encrypted) | Stateless validation. No session store needed for auth. |
   202тЖТ| Guest carts | Session-bound cart ID in cookie | No login required to browse and build a cart. |
   203тЖТ| Admin auth | Separate cookie (`ggh_admin_session`) | Isolated from customer sessions. Different expiry (8 hours). |

   231тЖТ| Update order status | тЬЧ | тЬЧ | тЬУ | тЬУ (assigned) |
   232тЖТ| ERP sync triggers | тЬЧ | тЬЧ | тЬУ | тЬЧ |
   233тЖТ
   234тЖТ### 4.5 OTP Authentication (Egypt-Specific)
   235тЖТ
   236тЖТGGH uses phone-based OTP instead of email/password. This is critical for the Egyptian market where many elder users don't use email.
   237тЖТ
   238тЖТ```
   239тЖТStep 1:  POST /api/v1/auth/otp/request
   240тЖТ         { "phone": "+201012345678" }
   241тЖТ         тЖТ SMS sent to phone
   242тЖТ         тЖТ Response: { "expires_in": 300 }  (5 minutes)
   243тЖТ
   244тЖТStep 2:  POST /api/v1/auth/otp/verify
   245тЖТ         { "phone": "+201012345678", "code": "1234" }
   246тЖТ         тЖТ On success: Set-Cookie with session
   247тЖТ         тЖТ On new customer: auto-create customer record
   248тЖТ         тЖТ Response: { "customer": { ... }, "is_new": true }
   249тЖТ```
   250тЖТ
   251тЖТ**OTP Rules:**
   252тЖТ
   253тЖТ| Rule | Value | Why |
   254тЖТ|---|---|---|
   255тЖТ| Code length | 4 digits | Elder-friendly. Easy to read and type. |
   256тЖТ| Expiry | 5 minutes | Security vs. usability balance. |
   257тЖТ| Max attempts | 3 per code | Prevent brute force. |
   258тЖТ| Resend cooldown | 60 seconds | Prevent SMS abuse. |
   259тЖТ| Max sends per day | 5 per phone number | Rate limit SMS costs. |
   260тЖТ

   263тЖТ## 5. Request Format
   264тЖТ
   265тЖТ### 5.1 Common Headers
   266тЖТ
   267тЖТ| Header | Required | Description |
   268тЖТ|---|---|---|
   269тЖТ| `Content-Type` | Yes (for body) | `application/json` |
   270тЖТ| `Accept-Language` | Optional | `ar` or `en`. Defaults to `ar` for Egyptian market. |
   271тЖТ| `X-Request-ID` | Optional | Client-generated UUID for tracing. Server generates if missing. |
   272тЖТ| `X-Idempotency-Key` | Optional | For safe retries on `POST` requests. Prevents duplicates. |
   273тЖТ
   274тЖТ### 5.2 Request Body Rules
   275тЖТ
   276тЖТ| Rule | Convention |
   277тЖТ|---|---|
   278тЖТ| Content type | `application/json` only. No form-encoded. |
   279тЖТ| Encoding | UTF-8 always |
   280тЖТ| Null handling | Omit null fields rather than sending `"field": null` |
   281тЖТ| Empty strings | Send `""` for cleared text fields. Never send `"null"` as a string. |
   282тЖТ| Money in requests | Integer (piastres). Client sends `11050` for EGP 110.50. |
   283тЖТ| Phone numbers | E.164 format: `+201012345678` |
   284тЖТ| Dates | ISO 8601: `2025-01-15T10:30:00Z` |
   285тЖТ| IDs | String (Medusa format): `"prod_01ABC"`, `"ord_01DEF"` |
   286тЖТ| Arrays | Always wrap in field: `{ "variant_ids": ["var_01", "var_02"] }` |
   287тЖТ| Bilingual fields | Send both or one: `{ "name_en": "Rice", "name_ar": "╪г╪▒╪▓" }` |
   288тЖТ
   289тЖТ### 5.3 Example Request
   290тЖТ
   291тЖТ```
   292тЖТPOST /api/v1/cart/items
   293тЖТContent-Type: application/json
   294тЖТAccept-Language: ar
   295тЖТX-Request-ID: req_uuid_abc123
   296тЖТ
   297тЖТ{
   298тЖТ  "variant_id": "variant_01JKL789",
   299тЖТ  "quantity": 2

   455тЖТ

   457тЖТ|---|---|---|
   458тЖТ| `200` | Successful retrieval or update | GET product, PATCH order |
   459тЖТ| `201` | Successful creation | POST new cart, POST checkout |
   460тЖТ| `204` | Successful deletion (no body) | DELETE cart item |
   461тЖТ| `400` | Validation error, malformed request | Missing required field, invalid phone format |

   506тЖТ### 7.5 Error Rules
   507тЖТ
   508тЖТ| Rule | Why |
   509тЖТ|---|---|
   510тЖТ| **Never expose stack traces** | Security risk. Confusing for users. |
   511тЖТ| **Always include Arabic message** | Elder users see Arabic error messages. |
   512тЖТ| **Always include `request_id`** | Support can trace the exact request in logs. |
   513тЖТ| **Never return raw Medusa errors** | BFF translates Medusa errors into GGH error codes. |
   514тЖТ| **Never return raw ERPNext errors** | Same reason. Wrap and translate. |
   515тЖТ| **5xx errors are generic** | "Something went wrong. Please try again." тАФ no internal details. |
   516тЖТ| **4xx errors are specific** | Tell the user exactly what to fix and how. |

   520тЖТ## 8. Pagination
   521тЖТ
   522тЖТ### 8.1 Pagination Style
   523тЖТ
   524тЖТGGH uses **page-based pagination**. Not cursor-based. Why?
   525тЖТ
   526тЖТ| Factor | Decision | Rationale |
   527тЖТ|---|---|---|
   528тЖТ| Page numbers in UI | Elder users understand "Page 3 of 8" | Cursor pagination is opaque and confusing |
   529тЖТ| Dataset size | Product catalog ~500тАУ2000 items | Not large enough to need cursor pagination |
   530тЖТ| Jump to page | Users need to jump to page 5 | Cursor can't do arbitrary page jumps |
   531тЖТ| SEO / sharing | URLs are shareable: `/products?page=3` | Cursor URLs are not human-readable |
   532тЖТ
   533тЖТ### 8.2 Pagination Parameters
   534тЖТ
   535тЖТ| Parameter | Type | Default | Max | Description |
   536тЖТ|---|---|---|---|---|
   537тЖТ| `page` | integer | 1 | тАФ | Page number (1-based) |
   538тЖТ| `per_page` | integer | 20 | 100 | Items per page |
   539тЖТ
   540тЖТ### 8.3 Request Example
   541тЖТ
   542тЖТ```
   543тЖТGET /api/v1/products?category=rice&page=2&per_page=20
   544тЖТ```
   545тЖТ
   546тЖТ### 8.4 Response Example
   547тЖТ
   548тЖТ```
   549тЖТ{
   550тЖТ  "data": [ ... 20 products ... ],
   551тЖТ  "meta": {
   552тЖТ    "request_id": "req_uuid_abc123",
   553тЖТ    "timestamp": "2025-01-15T10:30:00Z"
   554тЖТ  },
   555тЖТ  "pagination": {
   556тЖТ    "page": 2,
   557тЖТ    "per_page": 20,
   558тЖТ    "total": 47,
   559тЖТ    "total_pages": 3,
   560тЖТ    "has_next": true,
   561тЖТ    "has_prev": true
   562тЖТ  }
   563тЖТ}
   564тЖТ```
   565тЖТ
   566тЖТ### 8.5 Pagination Rules
   567тЖТ
   568тЖТ| Rule | Convention |
   569тЖТ|---|---|
   570тЖТ| Empty page | Return `200` with `data: []` and correct pagination metadata. Not `404`. |
   571тЖТ| `per_page` over max | Silently cap at 100. Don't error. |
   572тЖТ| `page` over total | Return `200` with `data: []` and `has_next: false`. |
   573тЖТ| Zero or negative values | Return `400` with `VALIDATION_INVALID_PARAMETER`. |

   577тЖТ## 9. Filtering & Sorting
   578тЖТ
   579тЖТ### 9.1 Filter Parameters
   580тЖТ
   581тЖТFilters are passed as query parameters. Only predefined filters are supported тАФ no arbitrary field filtering.
   582тЖТ
   583тЖТ```
   584тЖТGET /api/v1/products?category=rice&in_stock=true&price_max=50000&sort=price_asc
   585тЖТ```
   586тЖТ
   587тЖТ### 9.2 Common Filter Patterns
   588тЖТ
   589тЖТ| Filter | Type | Example | Description |
   590тЖТ|---|---|---|---|
   591тЖТ| `category` | string (slug) | `category=rice` | Filter by category slug |
   592тЖТ| `in_stock` | boolean | `in_stock=true` | Only show available items |
   593тЖТ| `price_min` | integer (piastres) | `price_min=5000` | Minimum price filter |
   594тЖТ| `price_max` | integer (piastres) | `price_max=50000` | Maximum price filter |
   595тЖТ| `search` | string | `search=╪г╪▒╪▓` | Full-text search (Arabic + English) |
   596тЖТ| `tag` | string | `tag=staple` | Filter by product tag |
   597тЖТ| `brand` | string | `brand=egyptian` | Filter by brand slug |
   598тЖТ
   599тЖТ### 9.3 Sort Parameters
   600тЖТ
   601тЖТ| Parameter | Values | Default |
   602тЖТ|---|---|---|
   603тЖТ| `sort` | `price_asc`, `price_desc`, `name_asc`, `name_desc`, `newest`, `popularity` | `popularity` |
   604тЖТ
   605тЖТSort values are human-readable strings, not raw column names. This prevents injection and makes the API self-documenting.
   606тЖТ
   607тЖТ### 9.4 Filter Rules
   608тЖТ
   609тЖТ| Rule | Convention |
   610тЖТ|---|---|
   611тЖТ| Unknown filters | Silently ignored. Don't error. |
   612тЖТ| Invalid filter values | Return `400` with `VALIDATION_INVALID_FILTER`. |
   613тЖТ| Multiple filters | AND logic. `category=rice&in_stock=true` = rice AND in stock. |
   614тЖТ| Empty result set | Return `200` with `data: []`. Not `404`. |
   615тЖТ| Search is special | `search` triggers full-text search across `name_en`, `name_ar`, `description_en`, `description_ar`, and `sku`. |
   616тЖТ

   716тЖТ### 10.2 Route Handler Pattern
   717тЖТ
   718тЖТEvery route handler follows this structure:
   719тЖТ
   720тЖТ```
   721тЖТimport { NextRequest, NextResponse } from 'next/server';
   722тЖТimport { withAuth } from '@/lib/api/middleware/auth';
   723тЖТimport { withErrorHandler } from '@/lib/api/middleware/error-handler';
   724тЖТimport { withRateLimit } from '@/lib/api/middleware/rate-limit';
   725тЖТimport { validateRequest } from '@/lib/api/validation';
   726тЖТimport { productSearchSchema } from '@/lib/api/schemas/product';
   727тЖТimport { productService } from '@/services/product';
   728тЖТ
   729тЖТexport const GET = withErrorHandler(
   730тЖТ  withRateLimit('product-list',
   731тЖТ    withAuth(async (request: NextRequest, { user }) => {
   732тЖТ      const params = validateRequest(request, productSearchSchema);
   733тЖТ
   734тЖТ      const result = await productService.search(params);
   735тЖТ
   736тЖТ      return NextResponse.json({
   737тЖТ        data: result.items,
   738тЖТ        meta: {
   739тЖТ          request_id: request.headers.get('x-request-id'),
   740тЖТ          timestamp: new Date().toISOString(),
   741тЖТ        },
   742тЖТ        pagination: result.pagination,
   743тЖТ      });
   744тЖТ    }, { optional: true })   // optional auth = guests can browse
   745тЖТ  )
   746тЖТ);
   747тЖТ```

  1369тЖТ## 18. Medusa API Proxy Rules
  1370тЖТ
  1371тЖТThe BFF acts as a smart proxy to Medusa. It never passes Medusa responses through unchanged.
  1372тЖТ
  1373тЖТ### 18.1 Translation Layer
  1374тЖТ
  1375тЖТ```
  1376тЖТBrowser Request          BFF Processing              Medusa Call
  1377тЖТтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФА          тФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФА              тФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФА
  1378тЖТGET /api/v1/products     тЖТ Validate params            тЖТ GET /store/products
  1379тЖТ                         тЖТ Add GGH price tiers         тЖТ  (with includes)
  1380тЖТ                         тЖТ Add delivery zone context
  1381тЖТ                         тЖТ Transform response
  1382тЖТ                         тЖТ Wrap in envelope
  1383тЖТ                         тЖР Return to browser
  1384тЖТ```
  1385тЖТ
  1386тЖТ### 18.2 Proxy Rules
  1387тЖТ
  1388тЖТ| Rule | Implementation |
  1389тЖТ|---|---|
  1390тЖТ| **Never expose Medusa URLs** | BFF maps GGH paths to Medusa paths internally. |
  1391тЖТ| **Translate error codes** | Medusa's `invalid_input` тЖТ GGH's `VALIDATION_REQUIRED_FIELD`. |
  1392тЖТ| **Inject GGH data** | Price tiers, delivery zones, and stock levels from `ggh` schema are merged into Medusa responses. |
  1393тЖТ| **Strip internal IDs** | Remove Medusa internal fields the frontend doesn't need. |
  1394тЖТ| **Add computed fields** | `isCasePack`, `minOrderQty`, `estimatedTotalAmount`. |
  1395тЖТ| **Cache product reads** | Product list and detail are cached in Redis with 5-minute TTL. |
  1396тЖТ| **Don't cache cart/order** | Cart and order data is always fresh. |
  1397тЖТ
  1398тЖТ### 18.3 Request Forwarding
  1399тЖТ
  1400тЖТ| BFF Path | Medusa Path | Method |
  1401тЖТ|---|---|---|

  1620тЖТ## 23. Localization & RTL in API
  1621тЖТ
  1622тЖТ### 23.1 Bilingual Field Convention
  1623тЖТ
  1624тЖТEvery user-facing text in API responses includes both languages:
  1625тЖТ
  1626тЖТ```
  1627тЖТ{
  1628тЖТ  "nameEn": "Egyptian Rice",
  1629тЖТ  "nameAr": "╪г╪▒╪▓ ┘Е╪╡╪▒┘К"
  1630тЖТ}
  1631тЖТ```
  1632тЖТ
  1633тЖТThe frontend selects the appropriate field based on the current locale. The API **never** decides which language to return.
  1634тЖТ
  1635тЖТ### 23.2 Why Both Fields, Always
  1636тЖТ
  1637тЖТ| Approach | Problem |
  1638тЖТ|---|---|
  1639тЖТ| Server-side locale switching | Requires two API calls for bilingual comparison. Breaks caching (different cache per locale). |
  1640тЖТ| Client-side language selection | One response serves both languages. Cache hit rate is higher. |
  1641тЖТ| Single field with language toggle | Elder users switching between languages lose their page state. |
  1642тЖТ
  1643тЖТ### 23.3 Accept-Language Header
  1644тЖТ
  1645тЖТThe `Accept-Language` header is used for **default** values only тАФ when a field is missing an Arabic translation, the API falls back to English. It never changes the response structure.
  1646тЖТ
  1647тЖТ| Header Value | Behavior |
  1648тЖТ|---|---|
  1649тЖТ| `ar` | If `nameAr` is null, fall back to `nameEn` in the `nameAr` field. |
  1650тЖТ| `en` | If `nameEn` is null, fall back to `nameAr` in the `nameEn` field. |
  1651тЖТ| Missing | Treat as `ar` (Egyptian market default). |
  1652тЖТ
  1653тЖТ### 23.4 Number Formatting
  1654тЖТ
  1655тЖТThe API returns raw numbers. The frontend formats:
  1656тЖТ
  1657тЖТ| Data | API Returns | Frontend Renders (AR) | Frontend Renders (EN) |
  1658тЖТ|---|---|---|---|
  1659тЖТ| Price | `27500` (piastres) | `┘в┘з┘е┘л┘а┘а ╪м.┘Е` | `EGP 275.00` |
  1660тЖТ| Quantity | `5` | `┘е` | `5` |
  1661тЖТ| Phone | `+201012345678` | `+20 101 234 5678` | `+20 101 234 5678` |
  1662тЖТ| Date | `2025-01-15T10:30:00Z` | `┘б┘е ┘К┘Ж╪з┘К╪▒ ┘в┘а┘в┘е` | `Jan 15, 2025` |

  1667тЖТ## 24. Elder-Friendly API Design
  1668тЖТ
  1669тЖТThese rules exist specifically because GGH serves Egyptian households where the primary user may be Om Ibrahim тАФ a 60-year-old woman using a phone on a 3G connection.
  1670тЖТ
  1671тЖТ### 24.1 Payload Minimization
  1672тЖТ
  1673тЖТ| Rule | Why |
  1674тЖТ|---|---|
  1675тЖТ| Product list returns only: id, names, thumbnail, price range, in stock | A list of 20 products with full detail is 200KB+ vs 20KB. On 3G, that's 2 seconds vs 20 seconds. |
  1676тЖТ| Product detail is a separate request | Only fetch when the user taps. |
  1677тЖТ| Cart summary, not full items, for header badge | Don't re-fetch the entire cart to show "3 items". |
  1678тЖТ| Image URLs, not base64 | Let the browser cache images natively. |
  1679тЖТ
  1680тЖТ### 24.2 Tolerance for User Mistakes
  1681тЖТ
  1682тЖТ| Rule | Implementation |
  1683тЖТ|---|---|
  1684тЖТ| **Auto-correct phone format** | `01012345678` тЖТ `+201012345678`. `00201012345678` тЖТ `+201012345678`. Don't reject; fix. |
  1685тЖТ| **Trim whitespace** | All string inputs are trimmed before validation. |
  1686тЖТ| **Quantity capping, not rejection** | If user requests 500 but only 340 available, offer 340 instead of erroring. |
  1687тЖТ| **Cart recovery** | Guest carts persist for 7 days. If the user closes the browser and returns, their cart is still there. |
  1688тЖТ| **Double-submit protection** | `X-Idempotency-Key` header prevents duplicate orders from accidental double-taps. |
  1689тЖТ| **Graceful degradation** | If ERP is down, show cached product data with a "prices may vary" notice. Don't show an error page. |
  1690тЖТ

  1709тЖТGGH uses Socket.io for real-time updates. WebSocket connections go through the same BFF.
  1710тЖТ
  1711тЖТ### 25.1 Connection
  1712тЖТ
  1713тЖТ```
  1714тЖТClient connects:
  1715тЖТ  io("/?XTransformPort=3003")
  1716тЖТ
  1717тЖТAuthentication:
  1718тЖТ  Send session cookie on handshake.
  1719тЖТ  Server validates JWT and joins user to their room.
  1720тЖТ```
  1721тЖТ
  1722тЖТ### 25.2 Event Catalog
  1723тЖТ
  1724тЖТ| Event | Direction | Room | Payload |
  1725тЖТ|---|---|---|---|
  1726тЖТ| `order:status_updated` | Server тЖТ Client | `customer:{customerId}` | `{ orderId, status, statusLabelEn, statusLabelAr, timestamp }` |
  1727тЖТ| `delivery:eta_updated` | Server тЖТ Client | `customer:{customerId}` | `{ orderId, eta, driverName, driverPhone }` |
  1728тЖТ| `stock:changed` | Server тЖТ Client | `product:{productId}` | `{ variantId, inStock, stockQuantity }` |
  1729тЖТ| `admin:order_new` | Server тЖТ Client | `admin:orders` | `{ orderId, customerName, totalAmount }` |
  1730тЖТ| `admin:sync_completed` | Server тЖТ Client | `admin:system` | `{ jobId, status, recordCount }` |
  1731тЖТ
  1732тЖТ### 25.3 WebSocket Rules
  1733тЖТ
  1734тЖТ| Rule | Convention |
  1735тЖТ|---|---|
  1736тЖТ| Never send sensitive data | No phone numbers, no addresses in WebSocket events. Use IDs and fetch detail via REST. |
  1737тЖТ| Reconnect with backoff | Client reconnects with exponential backoff: 1s, 2s, 4s, 8s, max 30s. |
  1738тЖТ| Missed events recovery | On reconnect, client fetches current state via REST. WebSocket is supplemental, not primary. |
  1739тЖТ| No WebSocket-only features | Every real-time feature has a REST fallback. |

  1743тЖТ## 26. API Changelog Convention
  1744тЖТ
  1745тЖТ### 26.1 Change Categories
  1746тЖТ
  1747тЖТ| Category | Description | Version Impact |
  1748тЖТ|---|---|---|
  1749тЖТ| **Added** | New endpoint, new optional field, new filter | No version bump |
  1750тЖТ| **Changed** | Field type change, field renamed, behavior change | Version bump |
  1751тЖТ| **Deprecated** | Field or endpoint marked for removal | No version bump, 6-month sunset |
  1752тЖТ| **Removed** | Field or endpoint deleted | Version bump |
  1753тЖТ| **Fixed** | Bug fix that doesn't change contract | No version bump |
  1754тЖТ
  1755тЖТ### 26.2 Changelog Format
  1756тЖТ
  1757тЖТEvery API change is recorded in `docs/api-changelog.md`:
  1758тЖТ
  1759тЖТ```
  1760тЖТ## 2025-01-15
  1761тЖТ
  1762тЖТ### Added
  1763тЖТ- `GET /api/v1/search` тАФ Full-text search endpoint
  1764тЖТ- `relatedProducts` field on product detail response
  1765тЖТ
  1766тЖТ### Changed
  1767тЖТ- `priceAmount` on variant is now always present (previously null for some case packs)
  1768тЖТ  тЖТ Migrate before upgrading to v2
  1769тЖТ
  1770тЖТ### Fixed
  1771тЖТ- `GET /api/v1/cart` now correctly calculates delivery fee for edge-case zones
  1772тЖТ```
  1773тЖТ
  1774тЖТ### 26.3 Breaking Change Policy
  1775тЖТ
  1776тЖТ| Policy | Rule |
  1777тЖТ|---|---|
  1778тЖТ| **Announce early** | Breaking changes are announced 6 months before activation. |
  1779тЖТ| **Coexist** | Old and new versions run simultaneously during migration. |
  1780тЖТ| **Sunset notice** | Deprecated endpoints return `Warning: 299 - "Deprecated: see /docs/api-changelog"` header. |
  1781тЖТ| **No silent breaks** | A field never changes meaning without a version bump. Adding a new field is never breaking. |
